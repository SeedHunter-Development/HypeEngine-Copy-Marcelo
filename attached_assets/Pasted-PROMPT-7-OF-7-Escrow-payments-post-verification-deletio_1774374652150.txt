PROMPT 7 OF 7: Escrow payments, post verification, deletion tracking + pricing impact
CONTEXT:
I have HypeEngine with PostgreSQL + Drizzle ORM. KOLs get matched to campaigns, receive AI-generated tweets with tracking links, and post them on Twitter. The system tracks clicks and conversions via the tracking pixel. Delivery scoring and the feedback loop already exist from previous prompts.

PROBLEM: Currently, when a KOL posts a tweet, credits are transferred immediately. There's no verification that the tweet stays live. KOLs can post, collect payment, then delete the tweet — destroying the campaign value. I need an escrow system with automated verification checks.

THE PAYMENT MODEL:
- Credits go into ESCROW when a KOL submits their post (not into the KOL's balance)
- System runs 4 automated checks at: 1 hour, 24 hours, 72 hours, and 7 days
- Each check scrapes the tweet URL to confirm it's still live and the tracking link is still present
- If the tweet survives all 4 checks: FULL payout released to KOL's balance, immediately withdrawable
- If the tweet is deleted OR the tracking link is removed at ANY point before day 7: FULL escrow returned to campaign budget, KOL gets nothing
- No partial payments. It's all or nothing.
- Deletion history is tracked on the KOL's profile and affects future matching, pricing, and visibility

DO NOT implement any account suspension detection or grace periods — keep it simple for MVP.

STEP 2 — ESCROW LOGIC
===========================================================

Create file: artifacts/hypeengine/lib/payments/escrow.ts

Function: createEscrow(postId: string): Promise<PostVerification>

Called when a KOL submits a post ("I Already Posted" or "Post Now"):
1. Load the post, campaign, and campaign_kol_match to get the agreed price
2. Calculate escrowAmount from the match's priceAgreed (or calculate via calculateKolPrice if not set)
3. Verify the campaign has enough remaining budget (credits - usedCredits >= escrowAmount)
4. Deduct escrowAmount from campaign's usedCredits (reserve it)
5. Set post.escrowStatus to "held"
6. Create a post_verifications row with:
   - escrowAmount
   - status = "verifying"
   - check1hAt = now + 1 hour
   - check24hAt = now + 24 hours
   - check72hAt = now + 72 hours
   - check7dAt = now + 7 days
   - All result fields = null
7. Create a transaction record: type "spend", description "Escrow held for post by @[handle]"
8. Return the verification record

DO NOT transfer any credits to the KOL's balance at this point.

Function: releaseEscrow(verificationId: string): Promise<void>

Called when all 4 checks pass:
1. Load the post_verification
2. Add escrowAmount to the KOL's user.credits balance
3. Set post.escrowStatus to "released"
4. Set post_verification.status to "passed"
5. Set post_verification.paidOutAt to now()
6. Update kol_profiles.totalPostsVerified += 1
7. Recalculate kol_profiles.postReliabilityRate
8. Create a transaction record: type "earn", description "Payout for verified post on [campaign title]"

Function: returnEscrow(verificationId: string, reason: string): Promise<void>

Called when any check fails:
1. Load the post_verification
2. Return escrowAmount to campaign budget (subtract from usedCredits)
3. Set post.escrowStatus to "returned"
4. Set post_verification.status to "failed"
5. Set post_verification.failedAt to now()
6. Set post_verification.failureReason to reason
7. Update kol_profiles.totalPostsDeleted += 1
8. Recalculate kol_profiles.postReliabilityRate
9. Create a transaction record: type "deposit", description "Escrow returned — post deleted/modified by @[handle]"

STEP 3 — VERIFICATION CHECKER
===========================================================

Create file: artifacts/hypeengine/lib/payments/verification.ts

Function: verifyPost(verificationId: string, checkType: "1h" | "24h" | "72h" | "7d"): Promise<VerificationResult>

This function:
1. Load the post_verification record
2. If status is already "failed", skip (don't re-check failed posts)
3. Use Apify to scrape the tweet URL:
   - Check if the tweet exists (returns content vs 404/not found)
   - If tweet exists, extract current engagement metrics (likes, retweets, replies, views)
   - Check if the tracking link URL (https://[DOMAIN]/r/[refCode]) is still present in the tweet text
4. Determine result:
   - Tweet exists AND tracking link present → result = "live"
   - Tweet does not exist / returns 404 → result = "deleted"
   - Tweet exists BUT tracking link is missing from text → result = "link_removed"
5. Store the result and engagement snapshot in the appropriate check fields:
   - If checkType is "1h": set check1hResult and check1hEngagement
   - If checkType is "24h": set check24hResult and check24hEngagement
   - etc.
6. If result is "deleted" or "link_removed":
   - Call returnEscrow(verificationId, `${result}_${checkType}`)
   - Return { passed: false, reason }
7. If result is "live" AND checkType is "7d":
   - This is the final check — ALL 4 checks passed
   - Call releaseEscrow(verificationId)
   - Return { passed: true, paidOut: true }
8. If result is "live" AND checkType is NOT "7d":
   - Return { passed: true, paidOut: false, nextCheck: next check type }

IMPORTANT: If the Apify scrape fails (API error, timeout, etc.), do NOT treat it as a deletion. Retry once after 5 minutes. If retry also fails, skip this check and try again at the next checkpoint. Log the error. Never penalize a KOL for your system's scraping failure.

Function: runPendingVerifications(): Promise<VerificationSummary>

This is the batch processor that runs all due checks:
1. Query post_verifications where status = "verifying"
2. For each, determine which checks are due (where check timestamp <= now AND result is null)
3. Run checks in order (1h first, then 24h, etc.) — never skip ahead
4. Process sequentially with 1-second delay between each (respect Apify rate limits)
5. Return summary: { checked, passed, failed, errors }

STEP 9 — POST-PAYOUT DELETION TRACKING
===========================================================

Even after the 7-day payout, you want to know if KOLs habitually delete campaign posts once they've been paid. No clawback, but it affects their reputation.

Create function: spotCheckPaidPosts(): Promise<SpotCheckResult>

This function:
1. Query posts where escrowStatus = "released" AND createdAt is between 8 and 30 days ago
2. Randomly sample 20% of these posts
3. For each sampled post, scrape the tweet URL via Apify
4. If the tweet no longer exists:
   - Update kol_profiles.postPayoutDeletions += 1
   - Recalculate postReliabilityRate to include post-payout deletions:
     postReliabilityRate = totalPostsVerified / (totalPostsVerified + totalPostsDeleted + postPayoutDeletions)
   - Log: "Post-payout deletion detected: @handle deleted tweet [url] on campaign [title]"
5. Return summary of spot check results

API endpoint: POST /api/admin/spot-check
→ Calls spotCheckPaidPosts()
→ Run weekly via cron (less critical than the verification checks)

STEP 11 — UPDATE DELIVERY SCORING
===========================================================

Modify artifacts/hypeengine/lib/analytics/delivery-score.ts:

When computing the delivery score for a KOL after campaign completion, use the engagement snapshots from post_verifications instead of re-scraping:
- check1hEngagement gives you initial viral spread
- check24hEngagement gives you peak engagement
- check72hEngagement gives you sustained engagement
- check7dEngagement gives you final numbers

This is better than a single scrape because you have the GROWTH CURVE:
- engagementGrowthRate = (check72hEngagement.likes - check1hEngagement.likes) / check1hEngagement.likes
- A post that gets 80% of its engagement in the first hour might be bot-boosted
- A post that grows steadily over 72h has genuine organic reach
- Factor this into the engagement delivery score as a quality signal

Add a new sub-dimension to delivery scoring:
ENGAGEMENT GROWTH PATTERN (weight 0.10, taken from existing engagement delivery weight which becomes 0.15):
- Calculate what % of total engagement arrived after the first hour
- Healthy: 40-70% of engagement comes after hour 1 (organic spread)
- Suspicious: >90% in first hour then flat (possible bot boost)
- Score: 1.0 for healthy growth pattern, 0.3 for suspicious front-loading
